# Code Review — RAG Prototype

## Overview

The codebase is a functional RAG MVP: ~450 lines of application code across 6 source files plus 4 scripts. It embeds user queries, retrieves relevant legal paragraphs from a pgvector-backed Supabase instance, and proxies the response through OpenRouter to a free-tier LLM. Traffic sign images are rendered via regex marker replacement instead of a Vision API.

The core architecture — ETL pipeline separate from the runtime, vector search delegated to Postgres via RPC, tenant isolation via RLS — is sound for a prototype. What follows are issues worth addressing, ordered by importance.

---

## 1. API key passed as query parameter (Medium severity)

**File:** `src/lib/ai-client.ts:29`

```ts
const res = await fetch(
  `https://generativelanguage.googleapis.com/v1beta/models/gemini-embedding-001:embedContent?key=${apiKey}`,
```

The Gemini API key is appended to the URL as a query string. Google [explicitly discourages this](https://cloud.google.com/docs/authentication/api-keys#using_api_keys) because query parameters are logged by proxies, load balancers, and CDNs. In a Vercel/Next.js deployment, the full URL — including the query string — appears in request logs.

**Fix:** Send the key as an HTTP header instead:

```ts
const res = await fetch(
  `https://generativelanguage.googleapis.com/v1beta/models/gemini-embedding-001:embedContent`,
  {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'x-goog-api-key': apiKey,
    },
    body: JSON.stringify({ ... }),
  }
);
```

The same issue exists in `scripts/diagnose.mjs:61` and `scripts/test_api.mjs:7`.

## 2. Duplicate retry logic with diverging behavior (Medium severity)

**Files:** `src/lib/ai-client.ts`, `scripts/ingest_universal.mjs`

Both files implement their own exponential backoff, but with different semantics:

| Aspect | `ai-client.ts` | `ingest_universal.mjs` |
|---|---|---|
| Retry trigger | Any non-ok response | Only HTTP 429 |
| Max retries | 3 | 5 |
| Base delay | 1000ms | 3000ms |
| Jitter | None | `Math.random() * 1000` |

The comment in both files acknowledges the duplication and warns to keep them in sync — but they already aren't. The ingest script is more resilient (retries on 429 only, adds jitter, longer base delay), while the runtime client retries on everything including permanent errors like 400/401. A 401 (bad key) will be retried 3 times with backoff before failing, which wastes time and masks the real problem.

**Fix:** Extract the shared retry logic into a standalone `.mjs` file that both the TypeScript runtime (via dynamic import or a small adapter) and the ESM scripts can use. At minimum, make the retry conditions consistent: only retry on 429 and 5xx, never on 4xx.

## 3. Tenant whitelist and UI out of sync (Low severity)

**File:** `src/app/api/chat/route.ts:11`, `src/app/page.tsx:69`

The server whitelists two tenants:

```ts
const VALID_TENANTS = new Set(['tenant-stvg', 'tenant-a']);
```

But the UI dropdown offers three:

```tsx
<option value="tenant-stvg">tenant-stvg (Deutsches Verkehrsrecht StVO/StVG)</option>
<option value="tenant-a">tenant-a (Galaxies)</option>
<option value="tenant-b">tenant-b (Empty)</option>
```

Selecting `tenant-b` always produces an error. If `tenant-b` is a placeholder for future use, consider either adding it to the whitelist (even if the database is empty — it would just return no results) or adding a small note in the UI. The current mismatch guarantees a broken path.

## 4. No response streaming (Documentation mismatch)

**File:** `src/app/api/chat/route.ts:93-103`

The OpenRouter call does not request streaming:

```ts
body: JSON.stringify({
  model: llm_provider,
  messages: messages
})
```

No `stream: true`, and the response is read with `genRes.json()`. This means the user sees nothing until the full LLM response is generated. For an MVP this is a reasonable simplification, but the original README's sequence diagram (step 92) claimed "Starts Streaming Markdown payload" which isn't accurate.

**Fix:** Either implement streaming (the infrastructure in `page.tsx` would need `ReadableStream` parsing) or document that responses are batched. Streaming is the more impactful UX improvement for a chat interface, but it adds meaningful complexity to both the API route and the client.

## 6. System prompt injection surface (Low severity)

**File:** `src/app/api/chat/route.ts:73-74`

The retrieved document content is interpolated directly into the system prompt:

```ts
const systemInstruction = tenant_id === 'tenant-stvg'
  ? `...\n\nKONTEXT:\n${contextText}`
  : `...\n\nCONTEXT:\n${contextText}`;
```

`contextText` comes from `matchedDocuments[].content`, which in turn comes from the database. If the source XML contained text that looks like prompt instructions (unlikely in legal documents, but possible), the LLM would interpret it as part of the system prompt. For an MVP with trusted data this is fine; for a system ingesting arbitrary user-supplied documents, the context should be delimited with explicit boundaries:

```ts
`...\n\nKONTEXT:\n"""\n${contextText}\n"""\n\nBeantworte die Frage ausschließlich anhand des obigen Kontexts.`
```

## 7. Anon key exposure context

**Files:** `src/app/api/chat/route.ts:6-7`, `.env.local`

The API route deliberately uses the anon key rather than the service role key:

```ts
const supabaseKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!;
```

This is the correct choice — it means even if the API route is compromised, the attacker can only read rows permitted by RLS. The comment on line 6 correctly notes this. However, the anon key is prefixed `NEXT_PUBLIC_`, which in Next.js means it will be inlined into the client-side JavaScript bundle. If `NEXT_PUBLIC_SUPABASE_ANON_KEY` isn't used anywhere in client components (it currently isn't — `page.tsx` doesn't import it), the prefix is misleading. Consider renaming to `SUPABASE_ANON_KEY` to avoid implying it's safe for client-side use, even if the RLS policy limits the damage.

## 8. Missing error handling in image regex

**File:** `src/app/page.tsx:12-24`

```tsx
const renderMessageContent = (text: string) => {
  const parts = text.split(/!\[.*?\]\((.*?)\)/g);
  return parts.map((part, index) => {
    if (index % 2 === 1) {
      return (
        <div key={index} ...>
          <img src={part} alt="Verkehrsschild" ... />
        </div>
      );
    }
    return <span key={index}>{part}</span>;
  });
};
```

This assumes every odd-indexed part is a valid image path. If the LLM hallucinates a markdown image with a malformed or missing URL, the `<img>` will render with a broken `src`. Consider adding a check that `part` starts with `/data/` or matches an expected pattern before rendering the image tag. Also, using array index as `key` is fragile — if the message content changes, React may reconcile incorrectly. A content hash or the URL itself would be more stable.

## 9. `cheerio` XML parsing silently degrades

**File:** `scripts/ingest_universal.mjs:83`

```js
const $ = cheerio.load(xmlData, { xmlMode: true, decodeEntities: false });
```

The `decodeEntities: false` means XML entities like `&auml;` are preserved as-is. This is intentional (the text later goes to an embedding model that handles Unicode), but it means the text stored in the database contains XML entities rather than the actual characters. If the text is ever displayed directly to a user without going through an HTML renderer, it will show `&auml;` instead of `ä`. This is unlikely in the current flow (the LLM normalizes it), but worth noting.

---

## Summary

The code is straightforward, readable, and correctly implements the core RAG pattern. The most actionable improvements are:

1. Move the API key from query string to header (security/logging hygiene)
2. Unify the retry logic so both runtime and ETL behave consistently
3. Fix the `tenant-b` UI/server mismatch
4. Update the layout metadata

None of these block the prototype from working. They matter most if this evolves beyond MVP stage.
